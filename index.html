What is seLe4n?

seLe4n is a micro-kernel written entirely in Lean 4 with machine-checked proofs, inspired by the seL4 architecture. It targets the Raspberry Pi 5 (ARM64 / BCM2712) as its first hardware platform.

Why a Micro-Kernel?

Micro-kernels minimize the trusted computing base by running only essential services in kernel space — scheduling, IPC, capability management, and memory control. Everything else (drivers, filesystems, networking) runs in user space. This reduces attack surface and makes formal verification tractable: every line of kernel code can be reasoned about mathematically.

Why Lean 4?

Lean 4 is both a general-purpose programming language and an interactive theorem prover. seLe4n leverages this duality: the kernel is executable code whose properties are machine-checked at compile time. There is no separate specification language, no trusted translation step, and no semantic gap between the model and the running code. If it compiles, the proofs hold.

How It Works: Pure Functional Kernel Transitions

In seLe4n, every kernel operation is a pure function from state to state. The entire kernel state — object stores, scheduler queues, capabilities, IPC queues, ASID tables — is captured in a single immutable SystemState record. Every system call takes the current state and returns a new state (or an error). There is no mutable global state, no side effects, and no hidden control flow.

This is made possible by the KernelM monad, defined as σ → Except ε (α × σ) — a state-transformer with explicit error handling. Every transition is deterministic: given the same input state and system call, you always get the same output state. This determinism is what makes formal verification not just possible, but practical.

What Makes It Different?

Unlike seL4 which uses C for implementation and Isabelle/HOL for proofs, seLe4n unifies implementation and proof in a single language. Each subsystem follows an Operations/Invariant split: executable transitions live in Operations.lean, and their machine-checked proofs live alongside in Invariant.lean. The type-checker enforces that every invariant holds across every operation.

Prelude & Machine

13 typed identifiers (ObjId, ThreadId, CPtr, Slot, ASID, VAddr, PAddr, etc.) each with Hashable + LawfulHashable instances. The KernelM monad, register files, memory regions, and timer configuration. Sentinel value 0 reserved for ObjId, ThreadId, ServiceId, and CPtr with identity theorems.

Object Model & State

8 kernel object types: Untyped, CNode, VSpaceRoot, TCB, Endpoint, Notification, IrqHandler, Service. The SystemState record aggregates all stores into a single immutable snapshot — HashMap-backed for O(1) access everywhere.

Scheduler

RunQueue backed by HashMap Priority (List ThreadId) with HashSet ThreadId membership index. Priority + EDF scheduling with domain-aware partitioning. flat_wf structural invariant. O(1) insert, remove, contains, and priority-bucketed candidate selection.

Capability & CDT

CNode slots backed by HashMap Slot Capability for O(1) operations. Node-stable Capability Derivation Tree with childMap : HashMap CdtNodeId (List CdtNodeId). Strict revocation error reporting via cspaceRevokeCdtStrict. Slot-reuse safety checks.

IPC & Dual-Queue

Intrusive dual-queue endpoints with sendQ/receiveQ and per-thread prev/next links for O(1) mid-queue removal. Message transfer via TCB.pendingMessage rendezvous. 14 preservation theorems. Legacy single-queue operations deprecated in v0.12.12.

Platform Layer

PlatformBinding typeclass defines runtime, boot, and interrupt boundary contracts. ExtendedBootBoundaryContract adds platform boot fields. Concrete bindings for a simulator and the Raspberry Pi 5 (BCM2712 memory map, GIC-400 constants, ARM64 config, 48-bit VA / 44-bit PA).

Key Innovations

seLe4n introduces architectural improvements across every subsystem, with a focus on O(1) data structures and machine-checked correctness. The WS-G performance portfolio eliminated all 14 audit findings.

O(1) Everywhere

Every hot-path data structure uses Std.HashMap or Std.HashSet: object stores, run queues, CNode slots, VSpace mappings, IPC queues, ASID tables, CDT child lookups, notification waiters, and service dependency graphs. All 13 typed identifiers have Hashable + LawfulHashable instances. 14 performance audit findings eliminated across 9 workstreams (WS-G1–G9). 400+ theorems re-verified after the data structure migration.

Priority-Bucketed Scheduler

The RunQueue is backed by HashMap Priority (List ThreadId) with a HashSet ThreadId membership index. Domain-aware EDF scheduling ensures threads only run in their assigned security domain. The flat_wf structural invariant is maintained across all operations. chooseBestInBucket reduces candidate selection from O(t) to O(k) where k is the bucket size. 13 bridge lemmas and 30+ IPC invariant proofs were migrated during the restructure.

Intrusive Dual-Queue IPC

Inspired by Linux kernel queue design. Endpoints maintain separate sendQ and receiveQ lists. Each TCB carries intrusive prev/next links enabling O(1) mid-queue removal without scanning. Messages flow through TCB.pendingMessage rendezvous with 14 preservation theorems (TPI-D08/D09) and 7 trace anchors. Sentinel ID rejection at IPC boundaries prevents ghost queue entries.

Capability Derivation Tree

Node-stable CDT with childMap : HashMap CdtNodeId (List CdtNodeId) for O(1) slot transfer and revocation. descendantsOf runs in O(N+E). Strict revocation via cspaceRevokeCdtStrict reports the first failure with full context rather than silently swallowing errors. Bidirectional childMapConsistent invariant with preservation proofs ensures the index stays synchronized with the underlying edge set.

N-Domain Information Flow

Parameterized security framework with two-dimensional labels (confidentiality + integrity). DomainFlowPolicy supports arbitrary domain counts and flow topologies with proven reflexivity and transitivity. GenericLabelingContext assigns domains to objects, threads, endpoints, and services. Per-endpoint flow policy overrides via EndpointFlowPolicy. The legacy 2×2 lattice embeds into a 4-domain linear lattice with proven semantics preservation. 15 non-interference theorems, 7-field ObservableState projection, precomputed HashSet-based observability for O(1) lookups.

Service Orchestration

First-class kernel service concept — a feature seL4 lacks entirely. Graph-based component lifecycle management with O(n+e) DFS cycle detection using HashSet ServiceId. Dependency-aware startup, shutdown, and restart with service health tracking. Acyclic policy enforcement prevents circular dependencies at the kernel level.

VSpace & Memory

Virtual address space management with HashMap VAddr PAddr mappings. Round-trip mapping theorems and universal noVirtualOverlap_trivial proof (trivially true via HashMap key uniqueness). Backend-independent design via VSpaceBackend adapter pattern. Untyped memory retype with watermark tracking, 10+ preservation theorems, and 6 negative-state tests. 48-bit VA / 44-bit PA address space for RPi5.

Composable Invariant Bundles

Each subsystem proves its own invariant bundle. Bundles compose upward: coreIpcInvariantBundle, ipcSchedulerCouplingInvariantBundle, lifecycleCompositionInvariantBundle, and vspaceInvariantBundle all roll up into proofLayerInvariantBundle, exposed as the unified apiInvariantBundle. A base-case theorem proves the bundle holds for the default (empty) state.

Inside the Code

Security Policy: Two-Dimensional Labels

SeLe4n/Kernel/InformationFlow/Policy.lean 
inductive Confidentiality where
  | low
  | high
  deriving Repr, DecidableEq

inductive Integrity where
  | untrusted
  | trusted
  deriving Repr, DecidableEq

structure SecurityLabel where
  confidentiality : Confidentiality
  integrity : Integrity

-- Combined policy: confidentiality flows up, integrity flows down
def securityFlowsTo (src dst : SecurityLabel) : Bool :=
  confidentialityFlowsTo src.confidentiality dst.confidentiality &&
    integrityFlowsTo dst.integrity src.integrity

-- Machine-checked: flow relation is transitive
theorem securityFlowsTo_trans
    (a b c : SecurityLabel)
    (h₁ : securityFlowsTo a b = true)
    (h₂ : securityFlowsTo b c = true) :
    securityFlowsTo a c = true := by
  cases a with
  | mk ac ai =>
      cases b with
      | mk bc bi =>
          cases c with
          | mk cc ci =>
              simp [securityFlowsTo] at h₁ h₂ ⊢
              exact And.intro
                (confidentialityFlowsTo_trans ac bc cc h₁.left h₂.left)
                (integrityFlowsTo_trans ci bi ai h₂.right h₁.right)

N-Domain Flow Policy

SeLe4n/Kernel/InformationFlow/Policy.lean 
-- Parameterized flow policy supporting arbitrary domain counts
structure DomainFlowPolicy where
  canFlow : SecurityDomain → SecurityDomain → Bool

-- Well-formedness: reflexive + transitive
def DomainFlowPolicy.wellFormed (p : DomainFlowPolicy) : Prop :=
  p.isReflexive ∧ p.isTransitive

-- Linear order: domain a can flow to domain b iff a.id ≤ b.id
def DomainFlowPolicy.linearOrder : DomainFlowPolicy :=
  { canFlow := fun src dst => decide (src.id ≤ dst.id) }

-- Proven well-formed
theorem DomainFlowPolicy.linearOrder_wellFormed :
    DomainFlowPolicy.linearOrder.wellFormed :=
  ⟨linearOrder_reflexive, linearOrder_transitive⟩

Security Model

Security is woven into every layer of seLe4n through formal information-flow control, capability-based access mediation, and a documented threat model.

Information Flow Control

seLe4n implements a parameterized N-domain information-flow framework. The system supports both the legacy 2×2 confidentiality/integrity lattice and arbitrary domain topologies via DomainFlowPolicy:

  • 15 non-interference theorems — 12 standalone + 3 enforcement bridges
  • 7-field ObservableState projection covering full kernel state
  • Precomputed observability via HashSet for O(1) projection lookups
  • 17 operations classified by enforcement level
  • Per-endpoint flow policies for fine-grained overrides
  • Legacy embedding theorem proves the generic system subsumes the original

Capability-Based Access

All resource access is mediated through capabilities stored in CNode slots. The capability derivation tree tracks all parent-child relationships:

  • O(1) slot operations via HashMap-backed CNodes
  • Strict revocation with explicit error reporting and context
  • Node-stable CDT with childMap HashMap index
  • Slot-reuse safety checks preventing dangling capabilities
  • Policy-checked wrappers: endpointSendDualChecked, cspaceMintChecked, serviceRestartChecked
  • Sentinel ID rejection at IPC boundaries prevents ghost queue entries

Threat Model & Security Boundaries

Protected Assets

  • Formal semantics and proof artifacts
  • Executable evidence (traces and tier scripts)
  • Supply-chain bootstrap infrastructure
  • Lean toolchain integrity

Threat Actors

  • External attacker: Public repo access, dependency injection
  • Upstream compromise: Altered installer delivery
  • Accidental insider: Secret commits, stale docs
  • Fork CI: Limited upload permissions

Security Controls

  • Expanded .gitignore secret/artifact denylist
  • SHA-256 verified elan installer downloads
  • Tier-3 anchor checks against CI workflow semantics
  • Non-blocking CodeQL security baseline

Verification & Testing

seLe4n employs a multi-tiered verification strategy combining Lean's type-checker with runtime testing, ensuring correctness from proof to execution.

Tier 0

Hygiene

Forbidden-marker scans ensure zero sorry and zero axiom across all production code. Documentation link validation, sync checks, and formatting enforcement. The first line of defense.

./scripts/test_tier0_hygiene.sh
Tier 1

Build

Full Lean compilation across 84 parallel build jobs. All 574 theorems are type-checked. Every invariant proof, every bridge lemma, every preservation theorem is verified by the Lean type-checker. If it compiles, the proofs hold.

./scripts/test_tier1_build.sh
Tier 2

Runtime

Trace sequence probes validate executable behavior against formal semantics. The negative-state suite tests malformed input rejection across IPC, capabilities, endpoints, and CSpace mutations. Information-flow runtime checks verify enforcement wrappers.

./scripts/test_tier2_negative.sh && ./scripts/test_tier2_trace.sh
Tier 3

Invariant Surface

Anchors every named theorem and invariant to the proof surface. Verifies that all documented claims in the Claim & Evidence Index are backed by actual code. Ensures no proof regressions across refactoring. The full correctness gate for PRs.

./scripts/test_tier3_invariant_surface.sh
Tier 4

Nightly

Determinism checks, experimental feature validation, and long-running invariant probes. Fixture-backed oracle testing. Catches subtle regressions that slip through faster tiers.

./scripts/test_nightly.sh

Getting Started

Whether you want to explore the kernel model, contribute proofs, or understand the architecture — here is how to get up and running.

1

Clone the Repository

git clone https://github.com/hatter6822/seLe4n.git
cd seLe4n
2

Set Up the Lean Environment

The setup script installs elan (the Lean version manager) with SHA-256 checksum verification and the pinned Lean 4.28.0 toolchain. 

./scripts/setup_lean_env.sh
3

Build & Verify

Build the entire kernel. All 574 theorems are checked during compilation across 84 parallel build jobs. Zero sorry, zero axiom. 

lake build
4

Run Tests

Start with the smoke test (minimum for PRs), then run the full suite for invariant work.

# Minimum PR requirement (Tiers 0-2)
./scripts/test_smoke.sh

# Full verification (Tiers 0-3)
./scripts/test_full.sh

# Quick CI check (Tiers 0-1 only)
./scripts/test_fast.sh

Contributing

seLe4n welcomes contributors. Before submitting a PR:

  • Read the Contributing Guide and Development Docs
  • Identify which workstream (WS-F5 through F8) your change addresses
  • Run ./scripts/test_smoke.sh at minimum; test_full.sh for theorem changes
  • Update invariants and theorems together with code changes
  • Keep PRs focused on one coherent unit of work
  • State transitions must have explicit success or failure outcomes

Roadmap

Development is organized into workstream portfolios and hardware binding milestones. 11 workstreams and 7 milestones are complete. Here is where the project stands.

M1–M7: Semantic Foundations Complete

Scheduler semantics, capability/CSpace operations, IPC seeds, waiting handshake, lifecycle/retype, capability composition, service graphs, architecture boundary assumptions, and initial audit remediation.

WS-B through WS-E: Audit Portfolios Complete

Four complete audit portfolios: WS-B comprehensive audit (11 workstreams), WS-C model/docs/maintainability (8 workstreams), WS-D test validity and proof gaps (6 workstreams), WS-E test/CI and kernel hardening (6 workstreams). All findings closed.

WS-F1–F4: Critical Audit Remediation Complete

Intrusive dual-queue IPC with message transfer verification. Untyped memory retype with watermark tracking. Information-flow completeness — 7-field ObservableState projection. Proof gap closure across all operations.

WS-G: Performance Optimization Complete

9 workstreams (G1–G9) migrating all hot paths to O(1) hash-based structures. 14 performance audit findings eliminated. Typed identifier Hashable instances, object store HashMap, ASID resolution table, RunQueue restructure, CNode slot HashMap, VSpace mapping HashMap, IPC dual-queue + notification, graph traversal optimization, and information-flow projection optimization. 400+ theorems re-verified.

WS-F5–F8: Refinement In Progress

F5: Model fidelity — badge bitmask, per-thread registers, multi-level CSpace.
F6: Invariant quality — tautology reclassification, adapter proof hooks.
F7: Testing expansion — oracle, probe, and fixture coverage.
F8: Cleanup — dead code removal, legacy/dual-queue resolution.

H3: Raspberry Pi 5 Platform Binding Upcoming

Structural foundation is in place: PlatformBinding typeclass, MachineConfig, VSpaceBackend, RPi5 stubs with BCM2712 memory map and GIC-400 constants. 48-bit VA / 44-bit PA address space. Blocked on WS-F closure.

H4 & IF-M5: Evidence Convergence Planned
