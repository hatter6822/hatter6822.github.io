What is seLe4n?

seLe4n is a micro-kernel written entirely in Lean 4 with machine-checked proofs, inspired by the seL4 architecture. It targets the Raspberry Pi 5 as its first hardware platform.

Why a Micro-Kernel?

Micro-kernels minimize the trusted computing base by running only essential services in kernel space. This architectural choice reduces attack surface and makes formal verification tractable — every line of kernel code can be reasoned about mathematically.

Why Lean 4?

Lean 4 is both a programming language and an interactive theorem prover. seLe4n leverages this duality: the kernel is executable code whose properties are machine-checked at compile time. No separate specification language, no trusted translation step.

What Makes It Different?

Unlike seL4 which uses C + Isabelle/HOL, seLe4n unifies implementation and proof in a single language. Every state transition is a pure function. The type-checker enforces that every invariant holds across every operation — with zero sorry and zero axiom.

Project Status

seLe4n is an active research project at version 0.12.15. The kernel model is feature-complete with IPC, scheduling, capabilities, memory management, and information-flow control. Hardware platform bindings for the Raspberry Pi 5 are in progress.

Key Innovations

seLe4n introduces several architectural improvements over traditional micro-kernel designs, with a focus on O(1) data structures and machine-checked correctness.

O(1) Everywhere

All hot paths use hash-based structures: object stores, run queues, CNode slots, VSpace mappings, IPC queues, ASID tables, and CDT child lookups. 14 performance audit findings eliminated.

Priority-Bucketed Scheduler

A RunQueue backed by HashMap Priority (List ThreadId) with domain-aware EDF scheduling. O(1) priority-bucketed enqueue, dequeue, and mid-queue removal.

Intrusive Dual-Queue IPC

Inspired by Linux kernel queue design: O(1) mid-queue removal via intrusive prev/next links on TCBs. Send and receive queues with full message transfer semantics.

Capability Derivation Tree

Node-stable CDT with a childMap HashMap for O(1) slot transfer and revocation. Strict error reporting prevents silent capability leaks.

N-Domain Information Flow

Parameterized security framework with two-dimensional labels. 15 non-interference theorems covering all kernel operations. Precomputed observability sets for O(1) projection lookups.

Service Orchestration

Graph-based component lifecycle management with O(n+e) DFS cycle detection. Dependency-aware startup, shutdown, and restart with service health tracking.

VSpace Abstraction

HashMap-based virtual address space management with round-trip mapping theorems. 48-bit VA / 44-bit PA address space for RPi5. Backend-independent design via VSpaceBackend adapter.

Untyped Memory Retype

Objects are carved from untyped memory with watermark tracking. 10+ preservation theorems and 6 negative-state tests ensure type-safe object creation.

Security Model

Security is not an afterthought — it is woven into every layer of seLe4n through formal information-flow control, capability-based access, and a rigorous threat model.

Information Flow Control

seLe4n implements a parameterized N-domain information-flow framework with two-dimensional security labels. The system provides:

  • 15 non-interference theorems — 12 standalone + 3 enforcement bridges
  • 7-field ObservableState projection covering full kernel state
  • Precomputed observability via HashSet for O(1) projection lookups
  • 17 operations classified by enforcement level

Capability-Based Access

All resource access is mediated through capabilities stored in CNode slots. The capability derivation tree (CDT) tracks all relationships:

  • O(1) slot operations via HashMap-backed CNodes
  • Strict revocation with explicit error reporting
  • Node-stable CDT with childMap indexing
  • Slot-reuse safety checks preventing dangling capabilities

Verification & Testing

seLe4n employs a multi-tiered verification strategy combining Lean's type-checker with runtime testing, ensuring correctness from proof to execution.

Tier 0

Hygiene

Forbidden-marker scans ensure zero sorry and zero axiom across all production code. Documentation link validation and sync checks.

./scripts/test_tier0_hygiene.sh
Tier 1

Build

Full Lean compilation across 84 parallel build jobs. All 574 theorems are type-checked. If it compiles, the proofs hold.

./scripts/test_tier1_build.sh
Tier 2

Runtime

Trace sequence probes, negative-state suite (malformed input rejection), and information-flow runtime checks. Validates executable behavior matches formal semantics.

./scripts/test_tier2_negative.sh
Tier 3

Invariant Surface

Anchors every named theorem and invariant to the proof surface. Ensures no proof regressions across refactoring. The full correctness gate.

./scripts/test_tier3_invariant_surface.sh
Tier 4

Nightly

Determinism checks, experimental feature validation, and long-running invariant probes. Catches subtle regressions that slip through faster tiers.

./scripts/test_nightly.sh

Getting Started

Whether you want to explore the kernel model, contribute proofs, or understand the architecture — here is how to get up and running.

1

Clone the Repository

git clone https://github.com/hatter6822/seLe4n.git
cd seLe4n
2

Set Up the Lean Environment

The setup script installs elan (the Lean version manager) and the correct Lean 4.28.0 toolchain with SHA-256 verified downloads.

./scripts/setup_lean_env.sh
3

Build & Verify

Build the entire kernel. All 574 theorems are checked during compilation.

lake build
4

Run Tests

Start with the smoke test for quick validation, then run the full suite.

# Quick validation (Tiers 0-2)
./scripts/test_smoke.sh

# Full verification (Tiers 0-3)
./scripts/test_full.sh

Contributing

seLe4n welcomes contributors. Before submitting a PR:

  • Read the Contributing Guide
  • Identify which workstream (WS-F5 through F8) your change addresses
  • Run ./scripts/test_smoke.sh at minimum
  • Update invariants and theorems together with code changes
  • Keep PRs focused on one coherent unit of work

Roadmap

seLe4n's development is organized into workstream portfolios. Here is the current status and what lies ahead.

Historical portfolio closure. Foundation work from v0.9.0 onward establishing core kernel model, initial proofs, and testing infrastructure.

Intrusive dual-queue IPC, untyped memory retype, CDT strict revocation, information-flow completeness, and proof gap closure. All high-priority audit findings resolved.

WS-G: Performance Optimization Complete

Migrated all hot paths to O(1) hash-based structures. 14 performance audit findings eliminated. 400+ theorems re-verified after data structure migration.

WS-F5–F8: Refinement In Progress

Model fidelity improvements (badge bitmask, per-thread registers), invariant quality refinement, testing expansion, and code cleanup.

H3: Hardware Binding Upcoming

Raspberry Pi 5 platform bindings with hardware-validated contracts. 48-bit VA / 44-bit PA address space. Runtime, boot, and interrupt boundary contracts via PlatformBinding typeclass.

IF-M5: Platform Integration Upcoming

Map model observability assumptions to RPi5 adapter obligations. Document unresolved security debt for side-channel follow-up.
