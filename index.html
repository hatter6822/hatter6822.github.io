What is seLe4n?

seLe4n is a capability-based microkernel whose kernel model is written entirely in Lean 4 with machine-checked proofs, complemented by safe Rust syscall wrappers for user-space programs. It targets the Raspberry Pi 5 (ARM64 / BCM2712) as its first hardware platform.

Why a Microkernel?

Microkernels minimize the trusted computing base by running only essential services in kernel space — scheduling, IPC, capability management, and memory control. Everything else (drivers, filesystems, networking) runs in user space. This reduces attack surface and makes formal verification tractable: every line of kernel code can be reasoned about mathematically.

Why Lean 4?

Lean 4 is both a general-purpose programming language and an interactive theorem prover. seLe4n leverages this duality: the kernel is executable code whose properties are machine-checked at compile time. There is no separate specification language, no trusted translation step, and no semantic gap between the model and the running code. If it compiles, the proofs hold.

How It Works: Pure Functional Kernel Transitions

In seLe4n, every kernel operation is a pure function from state to state. The entire kernel state — object stores, scheduler queues, capabilities, IPC queues, ASID tables — is captured in a single immutable SystemState record. Every system call takes the current state and returns a new state (or an error). There is no mutable global state, no side effects, and no hidden control flow.

This is made possible by the KernelM monad, defined as σ → Except ε (α × σ) — a state-transformer with explicit error handling. Every transition is deterministic: given the same input state and system call, you always get the same output state. This determinism is what makes formal verification not just possible, but practical.

What Makes It Different?

seLe4n unifies implementation and proof in a single language — no separate proof assistant, no translation layer, no semantic gap. Each subsystem follows an Operations/Invariant split: executable transitions live in Operations.lean, and their machine-checked proofs live alongside in Invariant.lean. The type-checker enforces that every invariant holds across every operation.

Prelude & Machine

13 typed identifiers (ObjId, ThreadId, CPtr, Slot, ASID, VAddr, PAddr, etc.) each with Hashable + LawfulHashable instances. The KernelM monad, register files, memory regions, and timer configuration. Sentinel value 0 reserved for ObjId, ThreadId, ServiceId, and CPtr with identity theorems.

Object Model & State

8 kernel object types: Untyped, CNode, VSpaceRoot, TCB, Endpoint, Notification, IrqHandler, Service. The SystemState record aggregates all stores into a single immutable snapshot — HashMap-backed for O(1) access everywhere.

Scheduler

133 theorems across 6 files. RunQueue backed by HashMap Priority (List ThreadId) with HashSet ThreadId membership index. Priority + EDF scheduling with domain-aware partitioning. flat_wf structural invariant. Operations split into Core, Preservation, and Selection. O(1) insert, remove, contains, and priority-bucketed candidate selection.

Capability & CDT

118 theorems across 5 modules. CNode slots backed by HashMap Slot Capability for O(1) operations. Node-stable Capability Derivation Tree with childMap : HashMap CdtNodeId (List CdtNodeId). Invariant proofs split into Authority (26 theorems), Defs (46 theorems), and Preservation (46 theorems). Strict revocation via cspaceRevokeCdtStrict.

IPC & Dual-Queue

Deeply modularized across 14 files with 223 theorems. Intrusive dual-queue endpoints split into Core, Transport, and WithCaps. 76 structural invariant proofs, 27 endpoint preservation theorems, and 16 capability transfer proofs. Operations decomposed into CapTransfer, Endpoint, and SchedulerLemmas.

Robin Hood Hash Map

Verified Robin Hood hash table implementation with 139 theorems across 7 modules. RHTable provides O(1) amortized insert, lookup, and erase with open addressing and linear probing. 30 bridge theorems connect the implementation to HashMap-compatible semantics. 56 preservation proofs and 40 lookup correctness theorems. Imported by Model.Object.Types — the verified foundation for all kernel hash-based data structures.

Information Flow

182 theorems across 9 deeply modularized files. Enforcement split into Soundness and Wrappers. Invariant proofs decomposed into Composition, Helpers, and Operations. 15 non-interference theorems with proven reflexivity and transitivity.

Service Orchestration

90 theorems across 4 files. Graph-based lifecycle with O(n+e) DFS cycle detection. Invariant proofs split into Acyclicity and Policy enforcement. Dependency-aware startup, shutdown, and restart.

Platform Layer

PlatformBinding typeclass defines runtime, boot, and interrupt boundary contracts. ExtendedBootBoundaryContract adds platform boot fields. Concrete bindings for a simulator and the Raspberry Pi 5 (BCM2712 memory map, GIC-400 constants, ARM64 config, 48-bit VA / 44-bit PA).

Rust Syscall Wrappers

Three no_std Rust crates provide ergonomic, fully safe wrappers around all 14 seLe4n syscalls. sele4n-types defines 14 #[repr(transparent)] newtype identifiers, a 34-variant KernelError enum, and bitmask-based AccessRights — all with #![deny(unsafe_code)]. sele4n-abi handles ARM64 register encoding/decoding, IPC buffer overflow, and MessageInfo bitfield packing — with exactly one unsafe block: the inline svc #0 instruction in trap.rs. sele4n-sys exposes typed wrappers for IPC (send/receive/call/reply), CSpace (mint/copy/move/delete), lifecycle (retype), VSpace (map/unmap with W^X enforcement), and service registry (register/revoke/query) — zero unsafe code.

Key Innovations

seLe4n introduces architectural improvements across every subsystem, with a focus on O(1) data structures and machine-checked correctness. The performance optimization portfolio eliminated all 14 audit findings.

O(1) Everywhere

Every hot-path data structure uses hash-based collections, backed by a verified Robin Hood hash map with 139 theorems: object stores, run queues, CNode slots, VSpace mappings, IPC queues, ASID tables, CDT child lookups, notification waiters, and service dependency graphs. All 13 typed identifiers have Hashable + LawfulHashable instances. All 14 performance audit findings eliminated. 400+ theorems re-verified after the data structure migration.

Verified Robin Hood Hash Map

A ground-up verified Robin Hood hash table across 7 modules. Open addressing with linear probing, RHTable provides O(1) amortized insert, lookup, erase, and resize with capacity preservation. 40 lookup correctness theorems, 56 preservation proofs, and 30 bridge theorems connecting the implementation to HashMap-compatible semantics (getElem?_insert_self, getElem?_erase_self, etc.). Imported by Model.Object.Types — the verified algebraic foundation for every kernel data structure.

Priority-Bucketed Scheduler

133 theorems across 6 files. The RunQueue is backed by HashMap Priority (List ThreadId) with a HashSet ThreadId membership index. Domain-aware EDF scheduling ensures threads only run in their assigned security domain. Operations decomposed into Core, Preservation, and Selection. The flat_wf structural invariant is maintained across all operations. Priority-bucketed candidate selection reduces complexity from O(t) to O(k) where k is the bucket size.

Intrusive Dual-Queue IPC

223 theorems across 14 deeply modularized files. Endpoints maintain separate sendQ and receiveQ lists. Each TCB carries intrusive prev/next links enabling O(1) mid-queue removal. Transport handles message transfer with 24 theorems. WithCaps adds capability transfer across IPC boundaries. 76 structural invariants, 27 endpoint preservation theorems, and call-reply-recv composition proofs.

Capability Derivation Tree

Node-stable CDT with childMap : HashMap CdtNodeId (List CdtNodeId) for O(1) slot transfer and revocation. descendantsOf runs in O(N+E). Strict revocation via cspaceRevokeCdtStrict reports the first failure with full context rather than silently swallowing errors. Bidirectional childMapConsistent invariant with preservation proofs ensures the index stays synchronized with the underlying edge set.

N-Domain Information Flow

182 theorems across 9 deeply modularized files. Parameterized security framework with two-dimensional labels (confidentiality + integrity). DomainFlowPolicy supports arbitrary domain counts and flow topologies with proven reflexivity and transitivity. GenericLabelingContext assigns domains to objects, threads, endpoints, and services. Enforcement split into Soundness and Wrappers; invariants into Composition, Helpers, and Operations. 15 non-interference theorems, 7-field ObservableState projection, precomputed HashSet-based observability for O(1) lookups.

Service Orchestration

90 theorems across 4 files. First-class kernel service concept with graph-based lifecycle management and O(n+e) DFS cycle detection using HashSet ServiceId. Invariant proofs split into Acyclicity and Policy enforcement. Dependency-aware startup, shutdown, and restart with service health tracking.

Architecture & Memory

123 theorems across 9 modules. Virtual address space with HashMap VAddr PAddr mappings and VSpaceBackend adapter pattern. TlbModel with flush-consistency proofs. RegisterDecode (18 roundtrip proofs) and SyscallArgDecode (28 determinism proofs) verify the hardware-software boundary end-to-end. 48-bit VA / 44-bit PA address space for RPi5.

Safe Rust Syscall Wrappers

Three no_std Rust crates wrap all 14 seLe4n syscalls with ergonomic, fully safe APIs. sele4n-types mirrors the Lean kernel’s typed identifiers and 34-variant error model with #![deny(unsafe_code)]. sele4n-abi handles ARM64 register marshalling with exactly one unsafe block — the svc #0 trap. sele4n-sys provides phantom-typed Cap<Obj, Rts> capabilities with compile-time rights tracking, W^X enforcement on VSpace mappings, and IPC buffer overflow handling for ARM64’s 4-register inline limit. Mock trap on non-AArch64 enables host-based testing.

Composable Invariant Bundles

Each subsystem proves its own invariant bundle. Bundles compose upward: coreIpcInvariantBundle, ipcSchedulerCouplingInvariantBundle, lifecycleCompositionInvariantBundle, and vspaceInvariantBundle all roll up into proofLayerInvariantBundle, exposed as the unified apiInvariantBundle. A base-case theorem proves the bundle holds for the default (empty) state.

Security Model

Security is woven into every layer of seLe4n through formal information-flow control, capability-based access mediation, and a documented threat model.

Information Flow Control

182 theorems across 9 deeply modularized files. seLe4n implements a parameterized N-domain information-flow framework supporting both the legacy 2×2 confidentiality/integrity lattice and arbitrary domain topologies via DomainFlowPolicy:

  • 15 non-interference theorems — 12 standalone + 3 enforcement bridges
  • 7-field ObservableState projection covering full kernel state
  • Enforcement split into Soundness and Wrappers
  • Invariant proofs decomposed into Composition, Helpers, Operations
  • Precomputed observability via HashSet for O(1) projection lookups
  • 17 operations classified by enforcement level
  • Per-endpoint flow policies for fine-grained overrides
  • Legacy embedding theorem proves the generic system subsumes the original

Capability-Based Access

All resource access is mediated through capabilities stored in CNode slots. The capability derivation tree tracks all parent-child relationships:

  • O(1) slot operations via HashMap-backed CNodes
  • Strict revocation with explicit error reporting and context
  • Node-stable CDT with childMap HashMap index
  • Slot-reuse safety checks preventing dangling capabilities
  • Policy-checked wrappers: endpointSendDualChecked, cspaceMintChecked, serviceRestartChecked
  • Sentinel ID rejection at IPC boundaries prevents ghost queue entries

Verification & Testing

seLe4n employs a multi-tiered verification strategy combining Lean's type-checker with runtime testing, ensuring correctness from proof to execution.

Tier 0

Hygiene

Forbidden-marker scans ensure zero sorry and zero axiom across all production code. Documentation link validation, sync checks, and formatting enforcement. The first line of defense.

./scripts/test_tier0_hygiene.sh
Tier 1

Build

Full Lean compilation across 114 parallel build jobs. All 1421 theorems are type-checked. Every invariant proof, every bridge lemma, every preservation theorem is verified by the Lean type-checker. If it compiles, the proofs hold.

./scripts/test_tier1_build.sh
Tier 2

Runtime

Trace sequence probes validate executable behavior against formal semantics. The negative-state suite tests malformed input rejection across IPC, capabilities, endpoints, and CSpace mutations. Information-flow runtime checks verify enforcement wrappers.

./scripts/test_tier2_negative.sh && ./scripts/test_tier2_trace.sh
Tier 3

Invariant Surface

Anchors every named theorem and invariant to the proof surface. Verifies that all documented claims in the Claim & Evidence Index are backed by actual code. Ensures no proof regressions across refactoring. The full correctness gate for PRs.

./scripts/test_tier3_invariant_surface.sh
Tier 4

Nightly

Determinism checks, experimental feature validation, and long-running invariant probes. Fixture-backed oracle testing. Catches subtle regressions that slip through faster tiers.

./scripts/test_nightly.sh

Getting Started

Whether you want to explore the kernel model, contribute proofs, or understand the architecture — here is how to get up and running.

1

Clone the Repository

git clone https://github.com/hatter6822/seLe4n.git
cd seLe4n
2

Set Up the Lean Environment

The setup script installs elan (the Lean version manager) with SHA-256 checksum verification and the pinned Lean 4.28.0 toolchain. 

./scripts/setup_lean_env.sh
3

Build & Verify

Build the entire kernel. All 1421 theorems are checked during compilation across 114 parallel build jobs. Zero sorry, zero axiom. 

lake build
4

Run Tests

Start with the smoke test (minimum for PRs), then run the full suite for invariant work.

# Minimum PR requirement (Tiers 0-2)
./scripts/test_smoke.sh

# Full verification (Tiers 0-3)
./scripts/test_full.sh

# Quick CI check (Tiers 0-1 only)
./scripts/test_fast.sh
5

Build Rust Syscall Wrappers

The rust/ workspace provides safe, no_std wrappers around all 14 seLe4n syscalls. Builds on both ARM64 (real trap) and x86_64 (mock trap for host-based testing). 

cd rust
cargo build
cargo test

Contributing

seLe4n welcomes contributors. Before submitting a PR:

  • Read the Contributing Guide and Development Docs
  • Identify which area of the project your change addresses
  • Run ./scripts/test_smoke.sh at minimum; test_full.sh for theorem changes
  • Update invariants and theorems together with code changes
  • Keep PRs focused on one coherent unit of work
  • State transitions must have explicit success or failure outcomes
